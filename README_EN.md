## Project Background

Based on sales data from an e‑commerce platform (orders, customers, products), this project builds a layered data warehouse (ODS → DWD → DWS → ADS) and focuses on common production pain points:

- Large scan volume caused by unreasonable partition strategies
- Query performance degradation due to small file explosion
- Performance differences among storage formats
- Long‑tail tasks caused by data skew

> This is a personal learning / job‑seeking project. All data has been desensitized or synthetically generated.

## Core Practices & Results

### 1. Partition Strategy Comparison

**Goal**: Compare the number of files and query performance for daily, monthly, and yearly partitions, and verify the effectiveness of partition pruning.

#### File Count Comparison
| Partition Granularity | File Count | Description |
|-----------------------|------------|-------------|
| Daily partition | 4018 | One file per date, severe small file issue |
| Monthly partition | 132 | One file per month, moderate |
| Yearly partition | 11 | One file per year, minimal |

#### Query Performance Comparison (date range `2020-01-01` ~ `2020-12-31`)
| Table | Partition Key | Avg Time (sec) | Scanned Partitions |
|-------|---------------|----------------|--------------------|
| Yearly partitioned (`fact_order_year_partitioned`) | year_key | 0.78 | 11 |
| Monthly partitioned (`fact_order_month_partitioned`) | month_key | 0.95 | 132 |
| Daily partitioned (`fact_order_day_partitioned`) | order_date | 1.24 | 4018 |

**Conclusion**: When designing a partition strategy, you must ensure that common query conditions directly use the partition key; otherwise, partitioning cannot improve performance.

### 2. Small File Compaction

**Issue**:  
- Since daily partitioning creates exactly one file per partition, it cannot demonstrate compaction of *multiple small files within a single partition*. An additional scenario is artificially simulated where three small files are created inside one partition to show the capability of `OPTIMIZE`.  
- In production, for the “one small file per partition” problem caused by overly fine‑grained partitioning, the solution is to redesign the partition granularity (e.g., switch to monthly partitioning), not to compact small files.

**Demonstration**: By appending data, we artificially create 3 small files in one partition; after compaction they become 1 file.

| Stage | File Count | Total File Size |
|-------|------------|-----------------|
| Before compaction | 3 | ~3 MB |
| After compaction | 1 | ~3 MB (size unchanged) |

![File Optimization Result](/Workspace/images/optimize_before_after.png "Code run result")

### 3. Storage Format Conversion (Delta vs ORC)

**Goal**: Compare storage size and query performance between Delta (Parquet) and ORC.  
**Method**: Run the same query three times and take the median execution time.

| Format | Table Type | File Size (MB) | Median Time (sec) |
|--------|------------|----------------|-------------------|
| Delta | Managed table | 74.75 | 0.91 |
| ORC   | External table | 72.00 | 2.69 |

**Observation**: The ORC external table performed worse.  
**Analysis**:  
> Delta managed tables benefit from Unity Catalog’s metadata caching, file skipping optimizations, and better data layout.  
> In production, Delta format is recommended for both ACID guarantees and performance gains.

![Storage Format Performance Comparison](/Workspace/images/format_comparison_1.png "Performance report")

### 4. Data Skew Handling

**Skew Detection**:  
- The top customer accounts for 3% of total orders (300,660 rows vs. median 379). Skew exists (largest key contributes 3% of total).

| Statistic | Order Count |
|-----------|-------------|
| Median | 379 |
| 90th percentile | 404 |
| 99th percentile | 425 |
| Max | 300,660 |

**Optimization Attempt**: Used two‑stage aggregation (adding random prefix to scatter keys). Because the total data volume is relatively small (10 million rows), the improvement was not obvious (execution time remained essentially the same). The execution plan showed that Spark’s optimizer automatically simplified the manual scattering logic.

**Conclusion**: With only 10 million rows, Spark AQE can handle the skew effectively; manual intervention does not bring noticeable benefits.

## How to Reproduce

1. **Environment Requirements**:
    - Recommended: Databricks workspace (free tier available).
    - Optional: Local Jupyter + PySpark (adjust file paths in the scripts if you wish).
2. **Data Preparation**:
    - Download the raw data (customer, product tables): `customer.csv`, `product.csv`
    - Generate raw order data: run `order_generator.py` to create synthetic order files (`orders_*.csv`) locally.
3. **Execution Order**:
    - Using the Databricks UI or command line, upload files to a Volume (skip if running locally).
    - Adjust directory parameters in the scripts to match your online catalog.
    - First run `01_setup_and_optimization.ipynb` to create tables, partitions, and perform small file compaction.
    - Then run `02_advanced_tuning.ipynb` for advanced tuning (partition comparison, format conversion, skew handling).
4. **Notebook Export**: Download the `.ipynb` files from the repository and import them into your Databricks workspace.

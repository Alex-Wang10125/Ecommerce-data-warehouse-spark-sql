## 项目背景

基于某电商平台的销售数据（订单、客户、产品），构建分层数仓（ODS→DWD→DWS→ADS），重点测试实际生产中的常见痛点：
- 分区策略不合理导致扫描数据量大
- 小文件膨胀导致查询性能下降
- 不同存储格式的性能差异
- 数据倾斜造成任务长尾

> 该项目为个人学习/求职作品，数据已脱敏或模拟生成。

## 核心实践与结果

### 1. 分区策略对比

**目标**：对比按天、按月、按年分区的文件数量与查询性能，验证分区裁剪的有效性。

#### 文件数量对比
| 分区粒度 | 文件数 | 说明 |
|---------|--------|------|
| 按天分区 | 4018 | 每个日期一个文件，小文件严重 |
| 按月分区 | 132   | 每个月份一个文件，适中 |
| 按年分区 | 11    | 每个年份一个文件，最少 |

#### 查询性能对比（日期范围查询 `2020-01-01` ~ `2020-12-31`）
| 表 | 分区键 | 平均耗时(秒) | 扫描分区数 |
|----|---------------|-------------|------------|
| 按年分区表 (`fact_order_year_partitioned`) | year_key | 0.78 | 11 |
| 按月分区表（`fact_order_month_partitioned`） | month_key | 0.95 | 132 |
| 按日分区表（`fact_order_day_partitioned`) | order_date | 1.24 | 4018 |

**结论**：在设计分区策略时，必须确保常用查询条件直接使用分区键，否则分区无法提升性能。

![分区查询性能对比](/images/partition_performance_1.png "查询运行结果")

### 2. 小文件合并

**问题**：<br>
    - 由于按天分区每个分区只有一个文件，无法演示“一个分区内多个小文件”的合并效果；此处额外模拟了一个分区内产生3个小文件的场景，用以展示 OPTIMIZE 的能力。<br>
    - 在生产环境中，对于这种分区过细导致的“每个分区一个小文件”问题，解决方法是重新设计分区粒度（如改为按月分区），而不是合并小文件。 <br>

**演示效果**：通过追加数据人为制造一个分区内的 3 个小文件，合并后变为 1 个文件。

| 阶段 | 文件数 | 文件总大小 |
|------|--------|-----------|
| 合并前 | 3 | 约 3 MB |
| 合并后 | 1 | 约 3 MB（大小不变） |

![文件优化结果](/images/optimize_before_after.png "代码运行结果")

### 3. 存储格式转换（Delta vs ORC）

**目标**：对比 Delta（Parquet）与 ORC 的存储大小和查询性能。<br>
**方法**：相同查询各运行3次，耗时取中位数。

| 格式 | 表类型 | 文件大小(MB) | 耗时中位数(秒) |
|------|--------|-------------|----------------|
| Delta | 托管表 | 74.75 | 0.91 |
| ORC  | 外部表 | 72.00 | 2.69 |

**观察**：ORC 外部表性能反而更差。<br>**分析**：<br>
>Delta 托管表享有 Unity Catalog 的元数据缓存、文件跳过等优化，且数据布局更优。<br>
>生产环境下推荐使用 Delta 格式以获得 ACID 和性能双重收益。

![存储格式性能对比](/images/format_comparison_1.png "性能报告")

---

### 4. 数据倾斜处理

**倾斜检测**：<br>
    - 订单量 Top 1 客户订单数占总量 3%（300,660 行 vs 中位数 379），存在倾斜（最大 key 占比 3%）。<br>


| 统计量 | 订单数 |
|--------|--------|
| 中位数 | 379 |
| 90% 分位数 | 404 |
| 99% 分位数 | 425 |
| 最大值 | 300,660 |

**优化尝试**：使用两阶段聚合（加随机前缀打散）。由于数据总量较小（1000 万行），优化效果不明显（耗时基本持平），执行计划显示 Spark 优化器自动简化了手动打散逻辑。

**结论**：由于数据总量仅千万级，Spark AQE 可有效处理，手动打散未带来明显收益。

## 如何复现

1. **环境要求**：
    - 推荐：Databricks 工作区（免费）。
    - 可选：本地 Jupyter + PySpark （需调整脚本文件路径）。
2. **数据准备**：
    - 下载原始数据（客户、产品表）： customer.csv, product.csv
    - 生成原始数据（订单表）：运行 `order_generator.py` ，本地生成模拟订单数据（orders_*.csv）。
3. **执行顺序**：
   - 打开工作区面板，使用 Databricks UI 或命令行上传文件到卷（本地运行可跳过）。
   - 调整脚本中的目录参数，以匹配您的在线 catalog。
   - 先运行 `01_setup_and_optimization.ipynb` 完成建表、分区、小文件合并。
   - 再运行 `02_advanced_tuning.ipynb` 进行高级调优（分区对比、格式转换、倾斜处理）。
4. **Notebook 导出**：可从仓库下载 `.ipynb` 文件，导入 您的Databricks 工作区。
